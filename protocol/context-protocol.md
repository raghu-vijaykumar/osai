# OSAI Context Protocol — Specification

**Version**: 1.0.0  
**Status**: Draft  
**Last Updated**: 2026-07-11

---

## 1. Overview

The OSAI Context Protocol is an application-agnostic, typed event protocol for capturing, storing, and querying digital activity context. Any application, connector, or service can publish events using this protocol. The protocol is:

- **App-agnostic** — no built-in assumptions about which applications produce or consume events
- **Typed** — every event type has a JSON Schema defining its payload shape
- **Versioned** — the protocol and individual event type schemas are independently versioned
- **Local-first** — events are published and queried locally; network transport is added in later protocol versions

### Design Principles

1. **Publish once, consume anywhere** — an event carries all the context needed at publish time; consumers do not need to reach back to the source
2. **Forward compatibility** — consumers ignore unknown fields in the payload; new optional fields never break existing consumers
3. **Self-describing** — every event carries its protocol version, type, and enough metadata to be understood without external schema lookup
4. **Permissioned** — every source must be registered and authorized to publish specific event types

---

## 2. Protocol Versioning

The protocol itself follows semver (MAJOR.MINOR.PATCH):

| Bump | When | Example |
|------|------|---------|
| MAJOR | Breaking change to the `ContextEvent` envelope — field removal, required field addition, transport change | `1.0.0` → `2.0.0` |
| MINOR | Backward-compatible addition — new optional field on the envelope, new transport option | `1.0.0` → `1.1.0` |
| PATCH | Clarification, documentation, no schema change | `1.0.0` → `1.0.1` |

**Current protocol version**: `1.0.0`

Every event carries its protocol version in the envelope. Consumers MUST check the protocol version and reject events with a MAJOR version higher than they support, returning error code `UNSUPPORTED_PROTOCOL_VERSION`.

Event type schemas are versioned independently (see section 7).

## 3. Event Envelope

### 3.1 Format

Every event is a JSON object conforming to this envelope:

```json
{
  "protocol_version": "1.0.0",
  "id": "0190a3f0-7c8a-7b00-9b00-000000000001",
  "source": "com.example.browser-extension",
  "type": "url.visited",
  "timestamp": "2026-07-11T14:30:00.123Z",
  "payload": { },
  "project": "optional-project-name",
  "session": "optional-session-id"
}
```

### 3.2 Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `protocol_version` | string (semver) | Yes | Protocol version used to produce this event. Set automatically by the SDK. |
| `id` | string (UUID v7) | Yes | Globally unique event identifier. Generated by the SDK at publish time. UUID v7 provides time-sortable IDs. |
| `source` | string | Yes | Identifier of the application/connector that produced the event. Format: reverse-domain (`com.example.myapp`). Must be registered in the permissions store. |
| `type` | string | Yes | Event type in dot-notation. See section 4 for naming conventions and section 5 for the core registry. |
| `timestamp` | string (ISO 8601) | Yes* | When the event occurred. If omitted by the publisher, the SDK assigns the current UTC time. |
| `payload` | object | Yes | Event-type-specific data. Validated against the registered JSON Schema for this type. |
| `project` | string | No | Project name or ID this event belongs to. Set explicitly by the source or inferred by the system. |
| `session` | string | No | Session ID grouping related events into a contiguous activity period. |

### 3.3 JSON Schema

The envelope itself has a JSON Schema:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://protocol.osai.app/envelope.json",
  "title": "ContextEvent",
  "type": "object",
  "required": ["protocol_version", "id", "source", "type", "timestamp", "payload"],
  "properties": {
    "protocol_version": {
      "type": "string",
      "pattern": "^\\d+\\.\\d+\\.\\d+$",
      "description": "Semver protocol version"
    },
    "id": {
      "type": "string",
      "format": "uuid",
      "description": "UUID v7 event identifier"
    },
    "source": {
      "type": "string",
      "pattern": "^[a-zA-Z][a-zA-Z0-9.]*$",
      "description": "Reverse-domain source identifier"
    },
    "type": {
      "type": "string",
      "pattern": "^[a-z][a-z0-9.]*(\\.[a-z][a-z0-9]*)+$",
      "description": "Event type in dot-notation"
    },
    "timestamp": {
      "type": "string",
      "format": "date-time",
      "description": "ISO 8601 UTC timestamp"
    },
    "payload": {
      "type": "object",
      "description": "Event-type-specific data"
    },
    "project": {
      "type": "string",
      "description": "Optional project association"
    },
    "session": {
      "type": "string",
      "description": "Optional session grouping identifier"
    }
  }
}
```

## 4. Event Type Naming

### 4.1 Conventions

Event types use lowercase dot-notation with reverse-domain namespacing:

```
{namespace}.{category}.{action}
```

Examples: `osai.url.visited`, `com.example.myapp.file.created`.

- **Namespaces**: `osai.*` is reserved for core OSAI event types. Third-party apps and connectors use their own reverse-domain namespace.
- **Categories**: Broad domain (`url`, `file`, `app`, `capture`, `note`).
- **Actions**: Past-tense verb (`visited`, `created`, `modified`, `focused`, `paused`).

### 4.2 Namespace Ownership

| Namespace | Owner | Purpose |
|-----------|-------|---------|
| `osai.*` | OSAI core | Built-in event types defined by this protocol |
| `custom.*` | User applications | User-defined or app-specific event types without a registered domain |
| `com.*`, `org.*`, `io.*`, etc. | Third-party apps | Registered reverse-domain namespaces |

Third-party applications MUST use their own reverse-domain namespace. The `osai.*` namespace is reserved and must not be used by external applications.

## 5. Core Event Types

### 5.1 Browser — `osai.url.*`

#### `osai.url.visited`

Published when a page is visited or a tab navigates.

```json
{
  "type": "url.visited",
  "payload": {
    "url": "https://example.com/page",
    "title": "Example Page",
    "domain": "example.com",
    "referrer": "https://google.com",
    "duration_ms": 45000,
    "tab_id": "tab_123",
    "content_preview": "This is the page content..."
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | string | Yes | Full URL of the visited page |
| `title` | string | Yes | Page title |
| `domain` | string | Yes | Extracted domain |
| `referrer` | string | No | Referrer URL, if available |
| `duration_ms` | number | No | Time spent on page in milliseconds (published on tab close or navigation away) |
| `tab_id` | string | No | Browser tab identifier for grouping |
| `content_preview` | string | No | First 500 characters of page content for context |

#### `osai.url.updated`

Published when the active tab's URL or title changes without full navigation (SPA route change).

```json
{
  "type": "url.updated",
  "payload": {
    "url": "https://example.com/new-route",
    "title": "New Route",
    "previous_url": "https://example.com/old-route",
    "domain": "example.com",
    "tab_id": "tab_123"
  }
}
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `url` | string | Yes | New URL |
| `title` | string | Yes | New page title |
| `previous_url` | string | No | Previous URL before navigation |
| `domain` | string | Yes | Extracted domain |
| `tab_id` | string | No | Browser tab identifier |

## 6. Validation Rules

### 6.1 Envelope Validation

All fields in the envelope are validated against the envelope schema before processing:

- `protocol_version` — must be a valid semver string; consumer rejects MAJOR version mismatches
- `id` — must be a valid UUID v7; SDK auto-generates if absent
- `source` — must be a registered source with permission for this event type
- `type` — must match a registered event type schema
- `timestamp` — if absent, SDK assigns current UTC time
- `payload` — validated against the registered JSON Schema for this type
- `project` and `session` — optional, no validation beyond type (string)

### 6.2 Payload Validation

Each event type's payload is validated against its registered JSON Schema at publish time. If validation fails, the event is rejected and an error is returned with details of the violation.

### 6.3 Timestamp Handling

- If `timestamp` is null, missing, or invalid, the SDK assigns the current UTC time
- Timestamps MUST be ISO 8601 with timezone indicator (Z for UTC)
- Consumers SHOULD accept timestamps in any ISO 8601 format and normalize to UTC

### 6.4 Forward Compatibility

- Consumers MUST ignore unknown fields in the payload (they are not validation errors)
- New optional fields are always backward-compatible
- Required fields can only be added in a MAJOR protocol version bump

## 7. Transport

### 7.1 Local IPC (v1.0.0)

In protocol version 1.0.0, transport is exclusively local:

- **Rust core → Tauri webview**: Tauri commands (invoke IPC)
- **Rust core → Node.js sidecars**: Named pipes (Windows) or Unix domain sockets (macOS/Linux)
- **Node.js sidecars ↔ Rust core**: Request-response via IPC with JSON serialization

No network transport is defined in v1.0.0. Network transport (HTTP, WebSocket) will be added in a future protocol version.

### 7.2 API Surface

The protocol exposes these operations via IPC:

| Operation | Method | Description |
|-----------|--------|-------------|
| `publish` | `publish(event)` | Publish a single event. Returns the generated event ID. |
| `publishBatch` | `publishBatch(events[])` | Publish multiple events atomically. Returns array of IDs. |
| `query` | `query(filters)` | Query events with filters. Returns paginated results. |
| `registerSource` | `registerSource(source)` | Register a new event source with permissions. |
| `registerSchema` | `registerSchema(type, schema)` | Register a JSON Schema for an event type. |

## 8. Permissions

### 8.1 Source Registration

Every event source must be registered before it can publish:

```json
{
  "id": "com.example.browser-extension",
  "name": "OSAI Browser Extension",
  "allowed_types": ["osai.url.*"]
}
```

The `allowed_types` field supports glob patterns: `osai.url.*` grants access to all `url` event types.

### 8.2 Permission Checking

At publish time, the system checks:

1. The source is registered — if not, reject with `SOURCE_NOT_FOUND`
2. The event type matches the source's `allowed_types` — if not, reject with `PERMISSION_DENIED`
3. The payload validates against the type's schema — if not, reject with `VALIDATION_ERROR`

## 9. Error Handling

### 9.1 Error Codes

| Code | HTTP Equivalent | Meaning |
|------|----------------|---------|
| `VALIDATION_ERROR` | 400 | Event payload failed schema validation |
| `PERMISSION_DENIED` | 403 | Source not authorized for this event type |
| `SOURCE_NOT_FOUND` | 404 | Source not registered |
| `UNSUPPORTED_PROTOCOL_VERSION` | 400 | Event protocol version not supported |
| `STORAGE_ERROR` | 500 | Storage layer failure |
| `INVALID_ENVELOPE` | 400 | Event envelope failed schema validation |

### 9.2 Error Response Format

All errors return a structured response:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Event payload failed schema validation",
    "details": [
      {
        "field": "payload.url",
        "rule": "type",
        "expected": "string",
        "actual": "number"
      }
    ]
  }
}
```

## 10. Extensibility

### 10.1 Defining Custom Event Types

Any application can define custom event types by:

1. Choosing a reverse-domain namespace (e.g., `com.myapp.*`)
2. Defining the event type name (e.g., `com.myapp.task.completed`)
3. Writing a JSON Schema for the payload
4. Registering the schema via `registerSchema(type, schema)`
5. Publishing events through the SDK

Custom event types are validated and stored identically to core types. No special handling is needed.

### 10.2 Namespace Reservation

To reserve a namespace for your application, submit a pull request to the protocol repository adding your namespace to the registry. This is optional but recommended to prevent collisions.
