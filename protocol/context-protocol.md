# OSAI Context Protocol ‚Äî Specification

**Version**: 1.0.0  
**Status**: Draft  
**Last Updated**: 2026-07-11

---

## 1. Overview

The OSAI Context Protocol is an application-agnostic event protocol for capturing, storing, and querying digital activity context. Any application, connector, or agent can publish events using this protocol. The protocol is:

- **App-agnostic** ‚Äî the same event types work across all applications; the `app` field identifies the source
- **Typed** ‚Äî every `event.action` pair has a JSON Schema defining its payload shape
- **Versioned** ‚Äî the protocol and individual event schemas are independently versioned
- **Log-first** ‚Äî events are appended to an ordered log on ingest; derived indexes are built asynchronously
- **Inbound + Outbound** ‚Äî the protocol captures actions observed *from* the world and actions initiated *by* the system

### Design Principles

1. **App-agnostic events** ‚Äî `file.opened` is the same event whether Chrome or VSCode fired it. The `app` field distinguishes the source.
2. **Forward compatibility** ‚Äî consumers ignore unknown fields in the payload
3. **Self-describing** ‚Äî every event carries its protocol version, app, event, action, and enough metadata to be understood without external lookup
4. **Log as source of truth** ‚Äî the append-only event log is the canonical record. Derived stores (SQLite, vector index) can be rebuilt from it.
5. **Inbound vs outbound** ‚Äî past-tense actions (`visited`, `modified`) describe observations. Base-form actions (`create`, `start`) describe commands or triggers. Both follow the same envelope and persist in the same log.

## 2. Protocol Versioning

The protocol follows semver (MAJOR.MINOR.PATCH):

| Bump | When | Example |
|------|------|---------|
| MAJOR | Breaking change to the envelope ‚Äî field removal, required field addition, transport change | `1.0.0` ‚Üí `2.0.0` |
| MINOR | Backward-compatible addition ‚Äî new optional field, new transport option | `1.0.0` ‚Üí `1.1.0` |
| PATCH | Clarification, documentation, no schema change | `1.0.0` ‚Üí `1.0.1` |

**Current protocol version**: `1.0.0`

Every event carries its protocol version. Consumers MUST reject events with a MAJOR version higher than they support, returning `UNSUPPORTED_PROTOCOL_VERSION`.

## 3. Event Envelope

### 3.1 Format

```json
{
  "protocol_version": "1.0.0",
  "id": "0190a3f0-7c8a-7b00-9b00-000000000001",
  "app": "com.google.Chrome",
  "event": "url",
  "action": "visited",
  "timestamp": "2026-07-11T14:30:00.123Z",
  "payload": {},
  "project": "osai",
  "session": "sess_abc"
}
```

### 3.2 Field Definitions

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `protocol_version` | string (semver) | Yes | Protocol version used to produce this event. Set automatically by the SDK. |
| `id` | string (UUID v7) | Yes | Globally unique event identifier. Generated by the SDK at publish time. |
| `app` | string | Yes | Source application identifier. Reverse-domain format (`com.google.Chrome`). Must be registered before first publish. |
| `event` | string | Yes | Generic event category ‚Äî what thing changed (`url`, `file`, `app`, `capture`, `note`, `session`, `project`, `action`). |
| `action` | string | Yes | The verb ‚Äî what happened to it. Past tense for inbound observations (`visited`, `modified`). Base form for outbound commands (`create`, `start`). |
| `timestamp` | string (ISO 8601) | Yes | When the event occurred. If omitted by the publisher, the SDK assigns current UTC time. |
| `payload` | object | Yes | Event-type-specific data. Validated against the registered JSON Schema for this `event.action`. |
| `project` | string | No | Project name or ID this event belongs to. |
| `session` | string | No | Session ID grouping related events into a contiguous activity period. |

### 3.3 Inbound vs Outbound

| Direction | Verb Form | Examples | Meaning |
|-----------|-----------|----------|---------|
| **Inbound** | Past tense | `visited`, `modified`, `focused`, `paused`, `launched` | Actions **received by** the platform ‚Äî observations captured from the world |
| **Outbound** | Base form | `create`, `start`, `end`, `schedule`, `archive`, `send` | Actions **initiated by** the platform/user/agent ‚Äî commands or triggers |

Both types follow the same envelope, validate against schemas, and persist in the same event log.

### 3.4 Directional Loop

```
Browser ‚Üí url.visited (inbound) ‚Üí Event Log     ‚Üí SQLite Index
                                                ‚Üí Agent queries via MCP
                                                ‚Üí Agent triggers session.start (outbound)
                                                ‚Üí Persisted to Event Log
                                                ‚Üí System processes ‚Üí session.started (inbound)
                                                ‚Üí UI reacts
```

An outbound action (`session.start`) is processed by the platform, which may emit a corresponding inbound event (`session.started`) as its result. This gives a complete audit trail: the command and its outcome are both recorded.

### 3.5 JSON Schema

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://protocol.osai.app/envelope.json",
  "title": "ContextEvent",
  "type": "object",
  "required": ["protocol_version", "id", "app", "event", "action", "timestamp", "payload"],
  "properties": {
    "protocol_version": { "type": "string", "pattern": "^\\d+\\.\\d+\\.\\d+$" },
    "id": { "type": "string", "format": "uuid" },
    "app": { "type": "string", "pattern": "^[a-zA-Z][a-zA-Z0-9.]*$" },
    "event": { "type": "string", "pattern": "^[a-z][a-z0-9]*$" },
    "action": { "type": "string", "pattern": "^[a-z][a-z0-9]*$" },
    "timestamp": { "type": "string", "format": "date-time" },
    "payload": { "type": "object" },
    "project": { "type": "string" },
    "session": { "type": "string" }
  }
}
```

## 4. Event Categories

### 4.1 Inbound Events (capture/observation)

| `event` | `action` | Payload Fields | Description |
|---------|----------|---------------|-------------|
| `url` | `visited` | `url`, `title`, `domain`, `referrer?`, `duration_ms?`, `tab_id?`, `content_preview?` | Page visited or tab navigated |
| `url` | `updated` | `url`, `title`, `previous_url?`, `domain`, `tab_id?` | SPA route change without full navigation |
| `file` | `created` | `path`, `filename`, `extension`, `size_bytes?`, `mime_type?`, `project_root?` | New file created |
| `file` | `modified` | `path`, `filename`, `extension`, `size_bytes`, `previous_size_bytes?`, `mime_type?`, `project_root?` | File content changed |
| `file` | `deleted` | `path`, `filename`, `extension`, `previous_size_bytes?`, `project_root?` | File removed |
| `file` | `opened` | `path`, `filename`, `extension`, `project_root?`, `app?` | File opened by an application |
| `app` | `focused` | `app_name`, `app_identifier`, `window_title?`, `bundle_id?`, `duration_ms?` | Application window gained focus |
| `app` | `idle` | `state` (idle/active), `duration_ms`, `idle_threshold_ms?` | User became idle or returned |
| `app` | `launched` | `app_name`, `app_identifier`, `bundle_id?` | Application launched |
| `app` | `quit` | `app_name`, `app_identifier`, `bundle_id?` | Application quit |
| `capture` | `paused` | `reason` (manual/privacy_mode/idle), `source` (tray-menu/shortcut/command-bar/auto), `duration_minutes?` | Capture paused |
| `capture` | `resumed` | `reason` (manual/timer_expired/app_restart), `source`, `paused_duration_ms?` | Capture resumed |
| `note` | `created` | `content`, `source` (command-bar/dashboard/agent-panel), `tags?` | Quick note created |
| `session` | `started` | `name?`, `project?`, `source?` | Session began (inbound confirmation) |
| `session` | `ended` | `duration_ms`, `event_count?`, `summary?` | Session ended |
| `project` | `created` | `name`, `source?` (auto-detected/manual) | Project detected or manually created |

### 4.2 Outbound Events (command/trigger)

| `event` | `action` | Payload Fields | Description |
|---------|----------|---------------|-------------|
| `note` | `create` | `content`, `tags?` | Request to create a quick note |
| `session` | `start` | `name?`, `project?` | Request to start a new session |
| `session` | `end` | `session_id` | Request to end a session |
| `project` | `create` | `name` | Request to create a project |
| `project` | `rename` | `project_id`, `new_name` | Request to rename a project |
| `project` | `archive` | `project_id` | Request to archive a project |
| `project` | `merge` | `source_project_id`, `target_project_id` | Request to merge two projects |
| `action` | `schedule` | `action_type`, `params`, `cron?`, `deadline?` | Schedule an action for later |
| `action` | `execute` | `action_type`, `params` | Execute an action immediately |
| `action` | `cancel` | `action_id` | Cancel a scheduled action |
| `event` | `sync` | `device_id?` | Request to sync events to cloud |
| `event` | `backup` | `destination?` | Request to create a backup |
| `event` | `export` | `format` (json/csv/pdf), `filters?` | Request to export data |

Each outbound event on the left may produce corresponding inbound events on the right when processed:

| Outbound | Inbound Result(s) |
|----------|-------------------|
| `session.start` | `session.started` (with the new session ID) |
| `project.create` | `project.created` (with the new project ID) |
| `action.schedule` | `action.scheduled` (with schedule confirmation) |
| `note.create` | `note.created` (with the persisted note) |

## 5. Source Registration

Before publishing, every source must register with the platform. No events are accepted from unregistered sources.

### 5.1 Registration Message

Sent once at connection time, before any events:

```json
{
  "protocol_version": "1.0.0",
  "id": "reg_abc123",
  "app": "com.myapp.extension",
  "event": "event",
  "action": "register",
  "timestamp": "2026-07-11T14:30:00.000Z",
  "payload": {
    "name": "My App Connector",
    "version": "1.0.0",
    "events": [
      { "event": "url", "actions": ["visited", "updated"] },
      { "event": "note", "actions": ["create"] }
    ]
  }
}
```

### 5.2 Registration Response

```json
{
  "status": "registered",
  "app": "com.myapp.extension",
  "granted_events": [
    { "event": "url", "actions": ["visited", "updated"] },
    { "event": "note", "actions": ["create"] }
  ],
  "rejected_actions": []
}
```

### 5.3 Registration Update

A source can update its registration at any time by sending a new registration message. The full list replaces the previous one.

### 5.4 Unregister

```json
{
  "action": "unregister",
  "payload": { "reason": "connector_disabled" }
}
```

## 6. Log-Based Ingestion Architecture

### 6.1 Pipeline

```
publish() ‚Üí Append-only Event Log (fast path)
                ‚Üì
       Background Consumer (sequential)
                ‚Üì
       Validation ‚Üí Schema check ‚Üí Permissions check
                ‚Üì
       SQLite Store ‚Üê‚îÄ Indexer ‚Üê‚îÄ Entity Extractor ‚Üê‚îÄ Embedder
                ‚Üì
       Queryable via SDK
```

- `publish()` appends to the log only ‚Äî returns immediately
- A background consumer processes the log sequentially, performing validation, extraction, embedding, and indexing
- The log IS the source of truth ‚Äî SQLite, vector index, and graph can be rebuilt from it at any time
- `query()` reads from the derived stores (may lag behind the log by milliseconds)

### 6.2 Properties

- **Ordered**: Events in the log are globally ordered by append time
- **Concurrent-safe**: Multiple writers append without coordination ‚Äî log handles it
- **Resilient**: If the consumer crashes mid-process, it resumes from its last checkpoint
- **Rebuildable**: `osai rebuild --from-log` recreates all derived stores

### 6.3 Checkpointing

The consumer records its position in the log after each successful batch. On restart, it resumes from the last checkpoint, reprocessing any events that were in-flight during the crash (idempotent ‚Äî events have unique IDs).

### 7.2 IPC Protocol Format

All communication over the named pipe / Unix socket uses **newline-delimited JSON (NDJSON)**. Each message is a single JSON object, terminated by `\n`. The receiver reads lines and parses each as JSON.

#### Request Format

```json
{
  "type": "<operation>",
  "id": "<uuid-v7>",
  "payload": {}
}
```

- `type` ‚Äî one of `register`, `publish`, `publish_batch`, `query`, `query_log`
- `id` ‚Äî client-generated, echoed back in the response (for correlating async replies)
- `payload` ‚Äî operation-specific fields

#### Response Format

```json
{
  "id": "<matching-request-id>",
  "ok": true,
  "result": {}
}
```

On error:

```json
{
  "id": "<matching-request-id>",
  "ok": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Event payload failed schema validation",
    "details": []
  }
}
```

#### Protocol Version Check

Before any operations, the client MAY send a lightweight handshake:

```json
{ "type": "handshake", "id": "h1", "payload": { "version": "1.0.0" } }
```

Response:

```json
{ "id": "h1", "ok": true, "result": { "version": "1.0.0", "server_id": "osai-core" } }
```

If the server's protocol version has a MAJOR mismatch, it returns `UNSUPPORTED_PROTOCOL_VERSION` and the client MUST disconnect.

### 7.3 Named Pipe (Windows) / Unix Socket (macOS/Linux)

| Property | Value |
|----------|-------|
| **Windows pipe name** | `\\.\pipe\osai` |
| **macOS/Linux socket path** | `/tmp/osai.sock` |
| **Fallback socket path** | `$XDG_RUNTIME_DIR/osai.sock` (Linux) |
| **Max message size** | 16 MB |
| **Encoding** | UTF-8 |
| **Concurrency** | Multiple concurrent clients; ordered by arrival |
| **Backlog** | SOMAXCONN (OS default) |
| **Permissions** | Socket file: `0700` (owner-only) on UNIX, pipe ACL-restricted on Windows |

The Rust core creates the pipe/socket on startup and listens for connections. If the socket file already exists (from a previous crash), the core deletes it before binding. On graceful shutdown, the socket/pipe is cleaned up.

### 7.4 Native Messaging Host (Browser Bridge)

Chrome and Firefox extensions cannot open arbitrary sockets. They communicate via the browser's `nativeMessaging` API, which spawns a helper executable.

#### Host Executable

- **Name**: `com.osai.native-host`
- **Location**: Installed by the OSAI desktop app at a platform-specific path
  - Windows: `%LOCALAPPDATA%\osai\native-host\com.osai.native-host.exe`
  - macOS: `/Applications/OSAI.app/Contents/Resources/native-host/com.osai.native-host`
  - Linux: `/opt/osai/native-host/com.osai.native-host`
- **Size target**: < 5 MB (could be a Rust binary or a Node.js script bundled with `pkg`)
- **Language**: Rust preferred (single binary, no runtime dependency)

#### Host Protocol

The browser communicates with the host via stdio using Chrome's native messaging framing:

```
[4 bytes: message length as uint32 little-endian] [JSON payload as UTF-8]
```

The host translates between the browser's framing and the OSAI named pipe / Unix socket:

```
Browser ‚îÄ‚îÄ[nativeMessaging framing]‚îÄ‚îÄ‚Üí Native Host ‚îÄ‚îÄ[NDJSON over pipe/socket]‚îÄ‚îÄ‚Üí Rust Core
```

The host's stdio-to-socket bridge:

```
1. Browser sends: [4-byte len]["register" message]
2. Host parses, forwards to socket as NDJSON
3. Host reads NDJSON response from socket, wraps in nativeMessaging framing, writes to stdout
4. Browser receives the framed response
```

#### Host Registration

The native messaging host must be registered with the browser via manifest files:

- **Chrome**: `chrome.nativeMessaging` requires a JSON manifest at a well-known path
  - Windows: `REG_SZ` under `HKEY_LOCAL_MACHINE\SOFTWARE\Google\Chrome\NativeMessagingHosts\com.osai.native-host`
  - macOS/Linux: JSON file at `/etc/opt/chrome/native-messaging-hosts/com.osai.native-host.json`
- **Firefox**: Similar manifest at the Firefox-specific path

The OSAI installer creates these manifests automatically.

### 7.5 Discovery

Connectors must discover how to reach the Rust core's IPC endpoint.

#### Deterministic Path

The simplest approach: well-known pipe/socket paths as defined in 7.3. Every connector uses the same path.

#### Runtime Info File (fallback)

For cases where the socket path might differ (e.g., per-user instances), the Rust core writes a small JSON file at startup:

```
~/.osai/runtime.json
```

```json
{
  "pid": 12345,
  "socket_path": "/tmp/osai.sock",
  "pipe_name": "\\\\.\\pipe\\osai",
  "protocol_version": "1.0.0"
}
```

The `@osai/protocol` SDK reads this file to discover the endpoint. If the file is missing or the PID is dead, the SDK returns `PLATFORM_NOT_RUNNING`.

### 7.6 Connection Lifecycle

#### Connect

```
1. @osai/protocol.connect() called
2. Read ~/.osai/runtime.json (fallback) or use well-known path
3. Open named pipe / Unix socket connection
4. Send handshake message (protocol version check)
5. On success: connection ready
6. On failure: retry up to 3 times with 1s backoff, then emit PLATFORM_NOT_RUNNING
```

#### Register

Every client **must** register before publishing. Registration is the first operation after connecting:

```
‚Üí { "type": "register", "id": "r1", "payload": { "app": "com.google.Chrome", "name": "Chrome Browser", "version": "1.0.0", "events": [{"event": "url", "actions": ["visited", "updated"]}] } }
‚Üê { "id": "r1", "ok": true, "result": { "app": "com.google.Chrome", "granted_events": [{"event": "url", "actions": ["visited", "updated"]}], "rejected_actions": [] } }
```

#### Disconnect & Reconnect

- If the socket disconnects (Rust core restarted), the SDK auto-reconnects with exponential backoff (1s, 2s, 4s, max 30s)
- The browser extension's native messaging host is re-spawned by the browser on the next `postMessage`
- On reconnect, the client must re-register (the server drops registrations on shutdown)

#### Buffering

When disconnected, the `@osai/protocol` SDK buffers publish requests in memory (up to 1000 events) and flushes them on reconnect in order. The buffer is capped ‚Äî if it overflows, the oldest events are dropped and a warning is logged.

## 8. Validation Rules

### 8.1 Envelope Validation

- `protocol_version` ‚Äî must be a valid semver string; consumer rejects MAJOR version mismatches
- `id` ‚Äî must be a valid UUID v7; SDK auto-generates if absent
- `app` ‚Äî must be a registered source; registration must grant this `event.action`
- `event` and `action` ‚Äî must match a registered event type schema
- `timestamp` ‚Äî if absent, SDK assigns current UTC time
- `payload` ‚Äî validated against the registered JSON Schema for this `event.action`

### 8.2 Payload Validation

Each `event.action` pair has a registered JSON Schema. If validation fails, the event is logged with an error marker so it can be re-processed if the schema is updated.

### 8.3 Forward Compatibility

- Consumers MUST ignore unknown fields in the payload
- New optional fields are always backward-compatible
- Required fields can only be added in a MAJOR protocol version bump

---

## 9. Permissions & Source Management

### 9.1 Source Registration

Every event source must register with:
- `app` ‚Äî unique reverse-domain identifier
- `name` ‚Äî human-readable name
- `version` ‚Äî source version string
- `events` ‚Äî list of `{event, actions[]}` this source is allowed to publish

### 9.2 Permission Checking

At publish time:
1. Source must be registered ‚Äî reject with `SOURCE_NOT_REGISTERED`
2. Event type must be in the source's granted list ‚Äî reject with `PERMISSION_DENIED`
3. Payload must validate against the schema ‚Äî reject with `VALIDATION_ERROR`

### 9.3 Registration Update

Sources can update registration at any time by sending a new registration. The update replaces the previous event list.

### 10.1 Error Codes

|------|---------|

| `PERMISSION_DENIED` | Source not authorized for this event.action |

| `UNSUPPORTED_PROTOCOL_VERSION` | Event protocol version not supported |
| `STORAGE_ERROR` | Storage layer failure |

## 11. Extensibility

### 11.1 Defining Custom Event Types

Applications can define custom `event.action` pairs:

1. Choose a reverse-domain `app` identifier
2. Define `event` and `action` names (lowercase, no dots)
3. Write a JSON Schema for the payload
4. Register the schema with the platform
5. Publish events through the SDK

### 11.2 MCP Compatibility

The Context Protocol is fully compatible with the Model Context Protocol (MCP, spec 025):

- MCP tools map to outbound events ‚Äî calling an MCP tool produces an outbound `event.action` in the log
- MCP resources map to query results from the derived store
- MCP prompts can reference events from the log

Any MCP action that triggers a system change produces an outbound event, which is persisted in the same log as all inbound capture events. This creates a unified, auditable record of both human and AI activity.

## Appendix B: Complete Example Flow

### 1. Source Registration

```
> { app: "com.myapp.ext", event: "event", action: "register", payload: { events: [{event: "url", actions: ["visited"]}] } }
< { status: "registered", granted_events: [...], rejected_actions: [] }
```

### 2. Inbound: Browser visits a page

```
> { app: "com.google.Chrome", event: "url", action: "visited", payload: { url: "...", title: "..." } }
< { id: "evt_001" }
```

### 3. Agent queries via MCP, decides to start a session

```
> { app: "com.osai.planner-agent", event: "session", action: "start", payload: { name: "Research", project: "osai" } }
< { id: "evt_002" }
```

### 4. Platform processes the start ‚Üí emits confirmation

```
> { app: "com.osai.platform", event: "session", action: "started", payload: { session_id: "sess_001", name: "Research" } }
< { id: "evt_003" }
```

### 5. All three events in the log

```
[0] { id: "evt_001", app: "com.google.Chrome",    event: "url",      action: "visited"  ... }
[1] { id: "evt_002", app: "com.osai.planner-agent", event: "session", action: "start"   ... }
[2] { id: "evt_003", app: "com.osai.platform",      event: "session", action: "started" ... }
```

The log captures both the observation (page visit) and the agent's action (start session) and its result (session started) ‚Äî everything in one place.
